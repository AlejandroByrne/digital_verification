# digital_verification — Local UVM via Docker + Verilator

Run UVM testbenches locally on macOS (Apple Silicon) with no commercial simulator.
Uses Verilator 5.046 + Accellera UVM 2020.3.1 + z3 SAT solver inside a Docker container.

## One-time setup

```bash
# 1. Make sure Docker Desktop is running
# 2. Build the shared Docker image (~2 min, only needed once)
make build
```

This creates a Docker image called `uvm-sim` containing:
- **Verilator 5.046** (built from source, ARM-native — no Rosetta emulation)
- **UVM 2020.3.1** (Accellera official reference implementation at `/opt/uvm-core/src/`)
- **z3 SAT solver** (for constrained randomization)

## Running tests

```bash
cd SA4
make smoke            # compile + run ha_smoke_test
make run T=ha_smoke_test   # re-run without recompiling (<1 sec)
```

See each subfolder's `Makefile` for the full list of targets.

## Where things live

### On your Mac (persists always)

| Path | What |
|---|---|
| `SA4/obj_dir/` | Compiled simulation binary (`Vtop`) + intermediate C++ files. Generated by Verilator from your `.sv`/`.svh` sources. Delete with `make clean`. |
| `SA4/*.sv`, `SA4/*.svh` | Your RTL and UVM testbench source files. |
| `Dockerfile` | Defines the `uvm-sim` Docker image. Shared by all subprojects. |

### Inside Docker (ephemeral container)

| Path | What |
|---|---|
| `/opt/verilator/` | Verilator installation (binaries, includes, share files). |
| `/opt/uvm-core/src/` | UVM library source. Referenced via `UVM_HOME` env var. |
| `/work/` | Mount point — maps to whichever subfolder you're running `make` from. |

### Docker named volume (persists across runs)

| Volume | What |
|---|---|
| `uvm-ccache` | C++ compiler cache. Stores compiled object files so that recompiles after source changes only rebuild what changed, not the entire UVM library. Without this, every compile would take ~15 min. With it, incremental compiles take ~30 sec. |

To inspect or clear the ccache volume:
```bash
# Check cache stats
docker run --rm -v uvm-ccache:/root/.cache/ccache uvm-sim ccache -s

# Clear the cache (forces full recompile next time)
docker volume rm uvm-ccache
```

## Compile times

| Scenario | Time |
|---|---|
| First compile ever (cold ccache, no obj_dir) | ~15 min |
| Recompile after changing a `.svh` file (warm ccache) | ~30 sec |
| Re-run with no source changes (`make run`) | <1 sec |
| `make recompile` (clean + full rebuild, warm ccache) | ~30 sec |

The first compile is slow because Verilator converts the entire UVM library (~100K lines of SystemVerilog) into ~2,400 C++ source files, then `g++` compiles each one. The ccache volume ensures this only happens once.

## Adding a new subproject (e.g. full adder, SA4 top)

1. Create the subfolder with your `.sv`/`.svh` files
2. Copy `SA4/Makefile` into it
3. Update the `SRC` variable to list your source files
4. Run `make smoke` (or whatever your test target is)

The same `uvm-sim` Docker image is shared across all subprojects. No need to rebuild it.

## Known limitations (Verilator vs commercial simulators)

- **`UVM_NO_DPI` is required** — Verilator's UVM support doesn't include DPI yet. `+UVM_TESTNAME` still works for test selection.
- **No waveform dump by default** — add `--trace` to `VFLAGS` in the Makefile if you need VCD output.
- **Experimental UVM support** — basic UVM components (agent, driver, sequencer, sequences, transactions, env, tests) work. Complex features (covergroups, register layer) may hit edges.
