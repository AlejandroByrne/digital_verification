# FP Multiplier UVM Testbench — Implementation Plan
## DO NOT READ — This is Claude's teaching guide

## DUT Summary
- Module: `fpmult_rtl #(P=8, Q=8)` — 16-bit custom FP format (1 sign + 8 exponent + 7 fraction = bfloat16-like)
- Bias = 2^(Q-1) - 1 = 127
- Clocked design with start/done handshake protocol
- Rounding modes: 0=RNE, 1=RTZ, 2=RD, 3=RU
- Special case handling: NaN, infinity, zero (subnormals NOT handled in multiplication path)
- oor_out[3:0] = {zero_flag, infinity_flag, nan_flag, subnormal_flag}

## Inputs/Outputs
- Inputs: rst_in_N, clk_in, x_in[15:0], y_in[15:0], round_in[1:0], start_in
- Outputs: p_out[15:0], oor_out[3:0], done_out

## Key Difference from Barrel Shifter
This is a CLOCKED design, but the computation is essentially COMBINATIONAL gated by a `go` register.

### CRITICAL: DUT Timing Bug / Quirk
The `done_out` signal is BROKEN for multi-transaction simulation:
- `done_1` is set to 1 in `always @(posedge go)` and NEVER cleared anywhere
- `done_2` is only ever set to 0 (when go==0), never set to 1
- `done_out = done_1 | done_2` → stays 1 PERMANENTLY after the first transaction
- Therefore: monitor CANNOT use `@(posedge done_out)` as trigger for 2nd+ transactions

### DUT Input Gating
Inputs are gated by `go`: `x_in_go = go ? x_in : 'x`
- When go=0, combinational outputs propagate 'x
- Outputs are only VALID while go=1 (and after #1 settle delay)
- `go` latches `start_in & rst_in_N` on posedge clk

### Correct Driver Protocol
1. Drive x_in, y_in, round_in on interface
2. Assert start_in = 1
3. Wait posedge clk → go latches high → inputs gate through
4. Wait posedge clk + #2 → combinational path settled (DUT has #1 internal delay)
5. **Outputs are now valid — monitor must sample HERE while go is still 1**
6. Deassert start_in = 0
7. Wait posedge clk → go goes low → inputs become 'x → outputs become 'x

### Correct Monitor Protocol
- Trigger on: `@(posedge dut_vi.clk iff dut_vi.start_in === 1'b1)` — detects START of each transaction
- Then wait: `@(posedge dut_vi.clk); #2;` — computation settling
- Sample all signals while they're still valid (go is still 1)
- Do NOT use done_out as trigger

## Coverage Model (THE TARGET)
### Coverpoints
1. cp_round_mode: 4 bins {RNE=0, RTZ=1, RD=2, RU=3}
2. cp_sign: 4 bins {pos_pos, pos_neg, neg_pos, neg_neg} — derived from x_in[15], y_in[15]
3. cp_x_class: 4 bins {zero, normal, infinity, NaN} (skip subnormal — DUT doesn't handle)
4. cp_y_class: 4 bins {zero, normal, infinity, NaN}
5. cp_oor: bins for each oor_out flag
6. cp_result_class: what class is the output?

### THE KEY CROSS (from resume)
cx_main: cross cp_round_mode, cp_sign, cp_x_class, cp_y_class
= 4 x 4 x 4 x 4 = 256 raw bins
Many auto-excluded (rounding mode irrelevant for NaN/inf/zero results)
Use ignore_bins for impossible/redundant combinations

### Additional Cross
cx_rounding_boundary: cross cp_round_mode, cp_sign — with constraint that
inputs are normal numbers near rounding boundaries (guard/round/sticky patterns)

## Operand Classes (IEEE 754 style)
- Zero: exponent == 0, fraction == 0
- Subnormal: exponent == 0, fraction != 0 (skip for this DUT)
- Normal: exponent != 0 AND exponent != all-1s
- Infinity: exponent == all-1s, fraction == 0
- NaN: exponent == all-1s, fraction != 0

## Phase 2: Stimulus Path (Steps 7-11)
### What student should propose:
- Driver with handshake protocol: drive inputs → pulse start → wait done
- Sequencer typedef
- Simple directed sequence (e.g., 1.5 x 2.0 = 3.0)
- Agent with active/passive support
- Base test class

### Key teaching points:
- Driver protocol is MORE COMPLEX than barrel shifter.
- CRITICAL: Cannot use `wait(done_out)` — done_out stays high after first txn (see timing bug above)
- Must keep start_in=1 during the sampling window so go stays 1 and outputs remain valid
- Deassert start_in AFTER monitor has sampled
```
forever begin
  seq_item_port.get_next_item(txn);
  // Drive inputs
  dut_vi.x_in     = txn.x_in;
  dut_vi.y_in     = txn.y_in;
  dut_vi.round_in = txn.round_in;
  // Assert start — go will latch on next posedge clk
  dut_vi.start_in = 1'b1;
  @(posedge dut_vi.clk);  // go latches high
  @(posedge dut_vi.clk);  // combinational logic settles (DUT has #1 internal delay)
  #2;                       // extra margin for settling
  // OUTPUTS ARE NOW VALID — monitor samples here
  dut_vi.start_in = 1'b0;  // deassert AFTER sampling window
  @(posedge dut_vi.clk);   // go goes low, outputs become 'x
  seq_item_port.item_done();
end
```
- Reset should be driven in the base test's run_phase BEFORE any sequences start
- The DUT's `go` signal latches on posedge clk when start_in & rst_in_N

### Known test values (P=8, Q=8, bias=127):
- 1.0 = 0_01111111_0000000 = 16'h3F80... wait P=8 means 7 fraction bits
  Actually: sign(1) + exponent(8) + fraction(7) = 16 bits
  1.0 = 0_01111111_0000000 = 16'h3F80
  Wait: 0_01111111_0000000 = 0011_1111_1000_0000 = 0x3F80
  2.0 = 0_10000000_0000000 = 0100_0000_0000_0000 = 0x4000
  1.5 = 0_01111111_1000000 = 0011_1111_1100_0000 = 0x3FC0
  3.0 = 0_10000000_1000000 = 0100_0000_0100_0000 = 0x4040

  1.5 x 2.0 should = 3.0
  x_in = 0x3FC0, y_in = 0x4000 → expected p_out = 0x4040

## Phase 3: Observation Path (Steps 12-15)
### What student should propose:
- Monitor that watches for start of each transaction (NOT done_out — it's broken)
- Scoreboard with behavioral reference model
- Analysis port chain: monitor → agent.ap → scoreboard + coverage

### Key teaching points:
- Monitor CANNOT use `@(posedge done_out)` — done_out stays 1 after first txn
- Must trigger on start_in going high, then wait for computation to settle
- Must sample BEFORE start_in is deasserted (while go=1, outputs valid)
- Monitor protocol:
```
forever begin
  // Detect start of a new transaction
  @(posedge dut_vi.clk iff dut_vi.start_in === 1'b1);
  // Wait for go to latch and combinational path to settle
  @(posedge dut_vi.clk);
  #2;  // DUT has #1 internal delay, +1 margin
  // Sample while outputs are still valid (go=1, start_in still=1)
  txn = fp_txn::type_id::create("mon_txn");
  txn.x_in     = dut_vi.x_in;
  txn.y_in     = dut_vi.y_in;
  txn.round_in = dut_vi.round_in;
  txn.p_out    = dut_vi.p_out;
  txn.oor_out  = dut_vi.oor_out;
  ap.write(txn);
end
```

### Reference Model (CRITICAL — this is the hard part):
The scoreboard's predict() function must:
1. Decompose x_in and y_in into sign, exponent, fraction
2. Check for special cases (NaN, inf, zero) — same logic as input_handler
3. Add implicit 1 to fractions
4. Multiply fractions (integer multiply of significands)
5. Add exponents, subtract bias
6. Normalize (shift product, adjust exponent)
7. Round according to mode using guard/round/sticky
8. Re-normalize if rounding caused overflow
9. Check for output saturation
10. Assemble final result

Alternative simpler reference: use $shortrealtobits conversion
But bfloat16 != float32, so we need custom logic.

SIMPLEST reference: use real arithmetic and manually convert
```systemverilog
function logic [15:0] predict(logic [15:0] x, logic [15:0] y, logic [1:0] round_mode);
  // Decode to real, multiply, re-encode to bfloat16 with specified rounding
endfunction
```

## Phase 4: Sequences (Steps 16-19)
### Sequences needed:
1. fp_rand_seq — constrained random, weighted distribution:
   - x_in and y_in: mostly normal numbers, some zeros, inf, NaN
   - round_in: equal weight across all 4 modes
   - sign: random

2. fp_directed_special_seq — hardcoded special cases:
   - NaN x normal = NaN
   - NaN x NaN = NaN
   - 0 x infinity = NaN
   - infinity x normal = infinity
   - infinity x infinity = infinity
   - 0 x normal = 0
   - 0 x 0 = 0
   - For each: test with both sign combinations

3. fp_directed_rounding_seq — rounding boundary conditions:
   - Values where guard=1, round=0, sticky=0 (exact tie for RNE)
   - Values where guard=1, round=1, sticky=0
   - Values where guard=1, round=0, sticky=1
   - Values where guard=0 (no rounding needed)
   - For each: test with all 4 rounding modes x both sign combos

4. fp_overflow_seq — exponent overflow/saturation:
   - Two large normal numbers whose product overflows
   - Numbers near max exponent

### Tests:
- test_directed: special + rounding + overflow sequences
- test_random: fp_rand_seq with N=500+ iterations
- test_full: directed first, then random

## Phase 5: Coverage (Steps 20-24)
### Coverage collector class:
```systemverilog
class fp_coverage extends uvm_subscriber #(fp_txn);
  // Classify operands
  function operand_class_e classify(logic [15:0] fp_val);
    // exponent all-0 + fraction 0 → ZERO
    // exponent all-0 + fraction != 0 → SUBNORMAL
    // exponent all-1 + fraction 0 → INFINITY
    // exponent all-1 + fraction != 0 → NAN
    // else → NORMAL
  endfunction

  covergroup fp_cg;
    cp_round_mode: coverpoint txn.round_in {
      bins rne = {2'd0};
      bins rtz = {2'd1};
      bins rd  = {2'd2};
      bins ru  = {2'd3};
    }
    cp_sign: coverpoint {txn.x_in[15], txn.y_in[15]} {
      bins pos_pos = {2'b00};
      bins pos_neg = {2'b01};
      bins neg_pos = {2'b10};
      bins neg_neg = {2'b11};
    }
    cp_x_class: coverpoint x_class {
      bins zero = {ZERO};
      bins normal = {NORMAL};
      bins infinity = {INFINITY};
      bins nan = {NAN};
    }
    cp_y_class: coverpoint y_class {
      bins zero = {ZERO};
      bins normal = {NORMAL};
      bins infinity = {INFINITY};
      bins nan = {NAN};
    }
    // THE KEY CROSS
    cx_main: cross cp_round_mode, cp_sign, cp_x_class, cp_y_class;
    // 4 x 4 x 4 x 4 = 256 raw bins
  endgroup
endclass
```

## Expected Interview Questions
1. "Why did you choose constrained-random over directed testing?"
   → Used both. CR for broad exploration, directed for rounding boundaries and special cases.

2. "How did you handle the reference model?"
   → Behavioral SV function that mirrors IEEE 754 multiply algorithm step by step.

3. "What was the hardest coverage bin to close?"
   → Rounding boundary conditions — specifically RNE tie-breaking with specific guard/round/sticky patterns combined with negative operands.

4. "How many transactions did it take to reach coverage closure?"
   → Directed covered special cases deterministically (~50-80 txns). Random filled cross bins (~500+ txns). Total ~600 to close.

5. "Why test rounding modes separately?"
   → Rounding is the most subtle part of FP — same inputs can produce different outputs depending on mode. The cross ensures every mode is exercised with every operand class and sign combination.

6. "What bugs did the testbench find?"
   → (Student should note any actual bugs found during development)

7. "Explain the Braun multiplier and why you chose it"
   → Array multiplier, fully combinational, no clock needed for the multiply step. Trade-off: area scales as O(n^2) but latency is predictable.

8. "What's the difference between normalization and renormalization?"
   → First normalization adjusts after the integer multiply. Renormalization adjusts after rounding (rounding can cause carry-out that requires shifting right and incrementing exponent).

## Common Pitfalls to Watch For
- Forgetting the start/done handshake in the driver (this isn't combinational like barrel shifter)
- Reference model disagreeing with DUT on rounding edge cases (the RNE tie-breaking is tricky)
- Monitor sampling before done_out — will get stale/wrong data
- The #1 delay in the DUT's done logic — simulation artifact, monitor needs to handle it
- oor_out encoding: {zero, infinity, nan, subnormal} = bits [3:1:0] — ordering matters
- exponent_adder saturates to all-1s on overflow (carry_out=1 → y_out = all-1s) — reference model must replicate this
